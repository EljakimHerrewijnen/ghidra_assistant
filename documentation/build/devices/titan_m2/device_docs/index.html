Titan M2

Docs about the Titan M2 fuzzer and implementation by the NFI.

Setup

Setup config.py file so that we can reuse the code made by Synacktiv. Create a file in the source/TitanM2EmulatorNFI called config.py and add the following contents:

GHIDRA_ASSISTANT_PATH="" # Irrelevant for now
TITANM2_EMULATOR_PATH = "/home/eljakim/Work/Opblaaskrokodil/opblaaskrokodil/source/TitanM2Emulator"

Follow the instructions in fuzzer_setup_label to build and install all Dependencies. Next, setup a python3 virtual environment and install the dependencies:

Setup python3 environment
$ python3 -m venv venv/
$ source venv/bin/activate
$ pip3 install -r requirements.txt

Install protobuf-compiler

$ sudo apt install protobuf-compiler libstdc++-12-dev

Build Clang 17

Build and install Clang 17.

git clone --depth=1 https://github.com/llvm/llvm-project.git
cd llvm-project
mkdir build
cd build
# compiler-rt for libclang_rt.fuzzer_no_main.a
cmake -DLLVM_ENABLE_PROJECTS="clang;compiler-rt" -DCMAKE_BUILD_TYPE=Release -G "Unix Makefiles" ../llvm
make

And install Clang17 system wide:

$ sudo make install

Copy llvm-project/build/lib/clang/17/lib/x86_64-unknown-linux-gnu/libclang_rt.fuzzer_no_main.a to the fuzzer_proto directory.

libprotobuf-mutator

Build and install libprotobuf-mutator

git clone https://github.com/google/libprotobuf-mutator
mkdir build
cd build
# DLIB_PROTO_MUTATOR_TESTING not needed but must faster and it needs internet
cmake .. -GNinja -DLIB_PROTO_MUTATOR_TESTING=OFF -DCMAKE_POSITION_INDEPENDENT_CODE=ON -DCMAKE_C_COMPILER=clang -DCMAKE_CXX_COMPILER=clang++ -DCMAKE_BUILD_TYPE=Debug
ninja
sudo ninja install

Next, build and install all other dependencies for unicorn and fuzzing:

Get and build unicorn
$ git clone git@github.com:unicorn-engine/unicorn.git
$ cd unicorn
$ git checkout e9c1c17f6df8f8f5da85ee80ad527452db5870ce
$ git apply ../unicorn.patch

Build
$ cd bindings/python/
$ python3 setup.py install

Run Emulator

You can run the emulator by running the GA_emulator file:

$ python3 GA_emulator.py

Update Protobufs

Protobufs should be backwards compatible.

git clone https://android.googlesource.com/platform/external/nos/host/generic
cd generic/generic/nugget/proto
find ./ -name "*.proto" -exec protoc {} --cpp_out=. --python_out=. \;
cp -rf nugget ../../../nugget_update
cd ../../../ && rm -rf generic/

nugget_update now contains the newest commands.

Development notes

Unicorn Patches

More patches were added to Unicorn to allow the dumping and restoration of new CSR registers. However, currently it is not possible to save and restore the state of the emulator.

Magisk

Download and extract the latest android studio and set it’s path in the ANDROID_SDK_ROOT environment variable.

export ANDROID_SDK_ROOT=~/android-studio
git clone https://github.com/topjohnwu/Magisk.git
cd Magisk/
git submodule update --init --recursive
./build.py ndk
./build.py binary

The pixel 6 uses android 13, where the ramdisk is no longer attached to the boot.img file. Instead this is in the init_boot.img file. This file was not included in the .zip files for the pixel 6, so these files were extracted from the pixel 7 .zip files.

Bad Weaver

Docs about when, after running the attack, the exploit fails and the weaver is in a bad state.

Zero Size:

Size of the weaver slot is set to 0. Confirmed Weaver slot will still move even when it is empty. Maybe we can overwrite it again?

It is not possible to move the corrupted weaver slot due to the size being 0. The weaver is never rewritten to flash. A dump of this state was made and loaded in a new emulator. This resulted in the emulator crashing with an actuall system exception.

Devices

Devices are emulated by creating a class for each device, which is based on the GSCDevice class:

class GSCDevice:

    BASE = None
    SIZE = None

    def __init__(self, gsc):
        self.gsc = gsc

    def read(self, address, size):
        raise Exception('must be implemented')

    def write(self, address, size, value):
        raise Exception('must be implemented')

Then, a hook is added when reading/writing from the device memory range.

def hook_device_access(self, handle, access, address, size, value, user_data):
    [..]

The devices are emulated by hooking writes to the MMIO regions:

self.mu.hook_add(UC_HOOK_MEM_READ|UC_HOOK_MEM_WRITE, self.hook_device_access, None, 0x40000000, 0x41000000)
self.mu.hook_add(UC_HOOK_MEM_READ|UC_HOOK_MEM_WRITE, self.hook_device_access, None, 0xe0000000, 0xe1000000)

These requests are then forwarded to the device to handle them.

SPI Device

Implementation of the SPI Device:

start: 0x40620000

The handling of the SPI data and setup is done in software, some events are simulated to create a valid execution state.

Support for generating an interrupt is added as part of the SPI device. The SPI device generates an interrupt when data is received. This interrupt is generated by:

  1. Interrupt in NVIC device is generated, the interrupt number is set here. NVIC is (probably) based on the ARM Cortex M3 Interrupt handling.

  2. Execution is transfered to the Interrupt code in the firmware.

If there are any flags, the spi_status of the class is updated with a hardcoded value, which is later passed on to the MMIO Status registers.

Next, the STATUS, STATUS_MASK, CMD, RX_SIZE and data is written to the regions in MMIO.

When execution is continued it will start at the interrupt handler.

NOS Calls

When injecting a NOS call, an interupt is generated by the SPI device.

First, the previously generated data is fetched from the SPI_RX_Buffer, next the command and data is transfered to the SPI device, which injects an interrupt and simulates the data flow.
